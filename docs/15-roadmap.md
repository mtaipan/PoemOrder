# 15. План развития (ТЗ)

Документ описывает целевой облик сайта PoemOrder и план доработок поверх
текущей реализации. Аудитория автора — мужчина 40 лет, пишет стихи на заказ
(свадьбы, юбилеи, признания, корпоративы). Большинство клиентов приходят с
телефона из соцсетей и Telegram.

Принцип: **не переделывать готовое**. Сначала фиксируем, что уже есть, затем —
только дельта.

## Что уже реализовано

**Стек:** Java 21 · Spring Boot 4 (Web MVC, Security, Data JPA, Validation,
Actuator) · PostgreSQL 16 · Flyway · Thymeleaf · Docker.

**Публичная часть:** главная (hero), портфолио + страница стиха, цены, отзывы
с модерацией, контакты, форма заказа.

**Админка `/admin`:** дашборд, CRUD стихов (Черновик/Опубликовано), заказы со
статусами (NEW → IN_PROGRESS → DONE / CANCELED), модерация отзывов, контакты,
настройки текстов. Пагинация по 20.

**Защита:** Spring Security, rate-limit публичных форм, honeypot.

**Telegram:** только **исходящие** уведомления (`TelegramNotifier.send()` →
`sendMessage` в фиксированный чат владельца). Входящих сообщений и интерактива
**нет** — настоящего бота ещё нет.

**Главные пробелы:** дизайн фронта, интерактивный бот, онлайн-оплата,
отслеживание статуса заказа клиентом, загрузка изображений, SEO.

## Целевой облик фронта (UX/UI)

### Дизайн-направление
- **Тон:** тёплый, литературный, «бутиковый». Не детский и не корпоративный.
- **Палитра:** фон айвори/кремовый (`#FBF8F2`), акцент — глубокий винный
  (`#7A2E2E`) или тёмно-изумрудный, тонкие золотистые детали (`#C8A96A`).
- **Типографика:**
  - заголовки и текст стихов — serif (Cormorant / PT Serif), крупный кегль,
    воздушный интерлиньяж (1.7+);
  - интерфейс/кнопки/меню — гротеск (Inter).
- **Mobile-first.** Сетка ломается в одну колонку, кнопки крупные, CTA «липкий»
  внизу экрана на мобильном.

### Структура главной (сверху вниз)
1. **Hero** — имя автора, строка-крючок, фото автора, кнопка «Заказать стих».
2. **Как это работает** — 3–4 шага: заявка → обсуждение → оплата → готовый стих.
3. **Поводы** — плитки: свадьба, юбилей, признание, день рождения, корпоратив.
   Клик по плитке → форма заказа с предзаполненным поводом.
4. **Избранные работы** — 3–6 карточек из портфолио.
5. **Отзывы** — карусель одобренных.
6. **Цены** — 3 пакета, кнопка «Выбрать» предзаполняет заказ.
7. **Финальный CTA + контакты/мессенджеры.**

### Элементы доверия
Фото автора, счётчик выполненных работ, «отвечаю в течение N часов»,
настоящие отзывы с именами/поводами.

### Доработки существующих страниц
- **Портфолио:** фильтр по поводу/тегам, теги на карточках.
- **Цены:** оформить как 3 пакета-карточки с кнопкой выбора.
- **Форма заказа:** добавить поля «повод», «дедлайн», «бюджет»,
  «предпочитаемый канал связи».

## Telegram-бот

Решение: **двусторонний бот — и для владельца, и для клиентов.** Текущий
`TelegramNotifier` (только исходящие) остаётся для нотификаций; добавляется приём
обновлений.

### Архитектура
- Эндпоинт `POST /telegram/webhook` (или long polling — webhook
  предпочтительнее на проде за HTTPS).
- Регистрация webhook при старте (`setWebhook`).
- Обработка `message` (команды, шаги диалога) и `callback_query` (инлайн-кнопки).
- Хранение состояния диалога: таблица `tg_session` (chat_id → state + черновик
  заказа в JSON) либо in-memory + БД-fallback.
- Разделение ролей: `chat_id` владельца из конфига → админ-команды; остальные →
  клиентский сценарий.

### Сценарий владельца
- Приходит уведомление о новой заявке/отзыве с **инлайн-кнопками**:
  «Взять в работу» / «Готово» / «Отменить» → меняют `OrderStatus` через
  `callback_query` без захода в админку.
- Команды: `/orders` (последние заявки), `/order N` (карточка), `/stats`.

### Сценарий клиента
- `/start` → приветствие + кнопки «Заказать стих», «Мой заказ», «Отзыв».
- **Заказ через бота:** повод → детали → дедлайн → контакт → создаётся `Order`
  (источник = `TELEGRAM`), клиент получает код заказа.
- **Статус:** «Мой заказ» по коду/номеру → текущий этап.
- **Выдача:** когда заказ `DONE` и есть готовый стих — бот присылает текст/файл.
- **Отзыв:** оформление отзыва → очередь модерации.

### Безопасность бота
- Проверка `secret_token` в заголовке webhook (`X-Telegram-Bot-Api-Secret-Token`).
- Rate-limit на chat_id.
- Эндпоинт webhook вне Spring Security form-login (отдельная permitAll-цепочка
  + проверка секрета).

## Модель данных — изменения

Новые/изменённые сущности (через Flyway, следующая версия миграции — `V11+`):

- **`orders`** — добавить: `code` (публичный код для отслеживания),
  `occasion`, `deadline`, `budget`, `contact_channel`, `source`
  (WEB / TELEGRAM), `paid` (bool), `amount`, `delivered_poem` (text/FK).
- **`tg_session`** — `chat_id`, `state`, `draft_json`, `updated_at`.
- **`payments`** (если онлайн-оплата) — `order_id`, `provider`, `external_id`,
  `status`, `amount`, `created_at`.
- **`poems`** — добавить теги/повод для фильтра портфолио (или таблица `tags`).
- **`media`/upload** — путь к изображению, привязка к poem/settings.

## Этапы и приоритеты

1. **Редизайн фронта** — типографика, hero, «как работает», поводы, адаптив.
   Максимальное влияние на конверсию.
2. **Бот владельца** — инлайн-кнопки статусов поверх текущих уведомлений
   (быстрый апгрейд, переиспользует `TelegramNotifier`).
3. **Статус заказа для клиента** — код заказа + страница `/order/{code}` +
   уведомление в ТГ при смене статуса.
4. **Онлайн-оплата** — ЮKassa, вебхук подтверждения, отметка `paid`.
5. **Загрузка изображений + SEO/OpenGraph + sitemap.**
6. **Бот для клиентов** — полный сценарий заказа/выдачи через бота
   (самый объёмный — в конце).

## Открытые вопросы

- Платёжный провайдер: ЮKassa, CloudPayments, или приём оплаты вручную?
- Бот: webhook (нужен публичный HTTPS-домен) или long polling?
- Нужен ли клиенту полноценный личный кабинет на сайте, или достаточно
  отслеживания по коду + Telegram?
- Хостинг готовых стихов: текст в БД или файл (PDF/изображение-открытка)?
